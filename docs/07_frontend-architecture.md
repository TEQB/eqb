# Frontend Architecture

**University Past Questions Platform**
Version 1.0 — Design & Architecture Phase

---

## 1. Frontend Philosophy

The frontend is built with one guiding principle: **get the student to the content in as few steps as possible**. Every screen is designed around navigation speed and clarity. Decorative UI is avoided. The hierarchy (Faculty → Department → Level → Course → Questions) is always visible and always one tap away.

## 3. Page-by-Page Specification

### Landing Page `/`
- Public, statically generated
- Hero: platform name, one-line description, "Get started" CTA
- Stats: total questions, total students (fetched at build time or ISR)
- No sidebar, clean minimal layout

### Register `/register`
**Components:** `RegisterForm`

Form fields in order:
1. Full name
2. University email (validated on blur: domain check)
3. Matric number
4. Faculty (select — fetched from DB)
5. Department (select — dynamically populated from faculty selection)
6. Current level (select — populated from `departments.available_levels`)
7. Password

On submit: Supabase signUp with metadata → redirect to `/register/verify`

OTP screen `/register/verify`:
- 6-digit OTP input (auto-focus, single input with splitting handled in JS)
- 10-minute timer with countdown
- "Resend OTP" button (active after 60s cooldown)
- On verify: redirect to `/dashboard`

### Login `/login`
- Email + password form
- "Forgot password?" → email OTP reset flow
- On login: middleware checks `is_locked` → redirect to dashboard or upload wall

### Dashboard `/dashboard`
**Data fetched server-side:**
- Student profile (name, department, current_level, is_locked, last_upload_at)
- Courses for current_level in student's department
- General courses
- Obligation status (days remaining)

**Layout:**
- Obligation banner (warning or hidden — never shown if compliant)
- Level tabs (100, 200, 300... based on `departments.available_levels`)
- General courses section (pinned, always visible)
- Course grid for selected level
- Each course card: code, title, question count, solution count

### Browse `/browse`
- Full Faculty → Department → Level → Course hierarchy in sidebar
- Main area: course grid for selected department/level
- Search bar at top: searches course code + title
- Level filter tabs

### Course Page `/course/[id]`
**Data:** course metadata + all published questions for that course

**Layout:**
- Course header: code, title, department, level, question count
- Filter bar: year (multi-select), semester (first/second/both), level
- Question list (sorted by year desc)
- Upload button (sticky on mobile — fixed bottom right)
- Empty state with upload CTA if no questions yet

### Question Detail `/question/[id]`
**Layout (stacked vertically):**
1. Question header: course, year, semester, level
2. File viewer (PDF or image, inline)
3. Download button
4. Flag button
5. Solutions section
   - "Add solution" button
   - Solution list (sorted by net votes desc)

### Upload `/upload`
**Two modes:**
1. Normal upload (from course page — course pre-filled)
2. Locked upload (from upload wall — student must select course themselves)

**Form fields:**
- File dropzone (drag + click, shows file name and type after selection)
- Course selector (searchable — code + title)
- Year (number input, 4 digits, 1990–current year)
- Semester (radio: First / Second)

**States:**
1. Idle — form ready
2. Uploading — file progress indicator
3. Reviewing — "Our AI is reviewing your upload..." with spinner
4. Success — "Published! View your question →"
5. Rejected — reason displayed + "Try again" button

**Upload Wall (when is_locked = true):**
- Full-screen, cannot navigate away
- Friendly message explaining the obligation
- Upload form embedded directly — same flow as above
- On success: redirect to dashboard

### Profile `/profile`
- Name, email, matric number (read-only)
- Department + level (level is editable)
- Upload history: list of student's own past questions + statuses
- Upload obligation status: last upload date, days remaining


## 5. State Management

No global state library (Redux, Zustand) is needed. State is managed at two levels:

**Server state** (data from Supabase):
- All listing and detail data is fetched in Server Components — no client state needed
- Mutations (upload, vote, flag) use Server Actions or client-side Supabase calls with `router.refresh()` to revalidate

**Local UI state** (Client Components only):
- Form state: `react-hook-form` with Zod validation
- Upload progress: local `useState`
- Filter selections: local `useState` with URL params (`useSearchParams`) for shareability
- Modal open/close: local `useState`

```typescript
// Pattern: Server Component fetches, Client Component handles interaction
// course/[id]/page.tsx (Server Component)
export default async function CoursePage({ params }) {
  const questions = await fetchQuestions(params.id)
  return (
    <>
      <CourseHeader course={...} />
      <QuestionFilters />           {/* Client Component */}
      <QuestionList questions={questions} />  {/* Server Component */}
    </>
  )
}
```

## 6. Navigation Architecture

### Desktop
- Persistent sidebar (240px) — always visible
- Sidebar tree: General Courses (pinned) → My Dept → Level accordion → Course list
- Header: logo, global search, notification bell, avatar menu

### Mobile
- Bottom navigation bar: Home, Browse, Upload, Profile
- Sidebar accessible via hamburger → Sheet (slides in from left)
- Upload button: floating action button (FAB) fixed bottom-right on course pages

### URL Structure
```
/                          → Landing
/login                     → Login
/register                  → Register
/dashboard                 → Student dashboard
/browse                    → Full browse hierarchy
/course/[id]               → Course question list
/question/[id]             → Question detail + solutions
/upload                    → Upload form (also upload wall)
/profile                   → Student profile
/admin/[secret]            → Admin dashboard (hidden)
```

## 7. Forms & Validation

All forms use `react-hook-form` + `zod` for schema validation.

### Registration schema
```typescript
const registerSchema = z.object({
  fullName: z.string().min(2).max(100),
  email: z.string().email().endsWith(process.env.NEXT_PUBLIC_UNIVERSITY_EMAIL_DOMAIN!),
  matricNumber: z.string().min(6).max(20),
  facultyId: z.string().uuid(),
  departmentId: z.string().uuid(),
  currentLevel: z.number().int().min(100).max(600),
  password: z.string().min(8),
})
```

### Upload schema
```typescript
const uploadSchema = z.object({
  file: z.instanceof(File)
    .refine(f => f.size <= 10 * 1024 * 1024, 'File must be under 10MB')
    .refine(f => ['application/pdf', 'image/jpeg', 'image/png'].includes(f.type), 'PDF or JPG/PNG only'),
  courseId: z.string().uuid(),
  year: z.number().int().min(1990).max(new Date().getFullYear()),
  semester: z.enum(['first', 'second']),
})
```

## 8. Performance Considerations

| Optimization | Implementation |
|---|---|
| Landing page is static | `generateStaticParams` + ISR every 24h |
| Course lists cached | `unstable_cache` with tag-based invalidation on new upload |
| PDF viewer lazy loaded | `dynamic(() => import('../components/PdfViewer'), { ssr: false })` |
| Images via Next.js `<Image>` | Automatic WebP conversion + sizing |
| Sidebar tree virtualized | Only expanded level's courses are in the DOM |
| Question list paginated | 20 per page with cursor-based pagination |
| Fonts preloaded | `next/font` with Inter — zero layout shift |
